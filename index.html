How a Server Suspension Taught Me Real-World DevOps

My ChemSus.in deployment burned through 2TB of bandwidth and got killed overnight. What I rebuilt — with a clear split between developer and client responsibilities — was far better than what I lost.

P
Pavan Kumar Ponnuri
AI Native Developer
~ 10 min read

There's a particular kind of dread that hits when you open your browser, type in your project's URL, and get nothing back. No error page. No timeout spinner. Just emptiness. That's exactly what happened to chemsus.in — a project I'd spent weeks building and deploying — when the server hosting it was suspended without warning.

ChemSus is a full-stack Node.js application backed by SQLite, built for ChemSus Technologies — an Indian specialty chemicals company developing sustainable, biomass-derived products like levulinic acid and 5-HMF. I had deployed it on a RackNerd VPS, configured Nginx as a reverse proxy, set up SSL via Let's Encrypt, wired the whole thing through PM2, and had it running in production. It was live. It was working. And then one morning, it simply wasn't.

The cause was embarrassingly simple: I had burned through 2 terabytes of network bandwidth. The server plan had a hard cap, and I'd blown right past it. RackNerd suspended the instance and sent an offer — pay $7 to extend the bandwidth. But the seven dollars wasn't the real problem. The architecture decision that led me to consume 2TB in the first place was.

• • •

The Mistake: One Server, Two Jobs

Here's what I was doing wrong, and it's a mistake many early-stage developers make: I was using the same server for both production and active development. The production app was live on chemsus.in, serving real visitors. And on that exact same machine, I was writing code manually and through Claude CLI, pulling dependencies, and frequently installing and testing packages.

Every npm install, every git pull, every Claude CLI session was eating into the bandwidth. Development tools are chatty — they download packages, fetch models, stream responses back and forth. On a local machine, that traffic is invisible. On a metered VPS with a hard bandwidth limit, it's a ticking time bomb.

I wasn't just running a website on that server. I was living on it — coding, debugging, deploying, all on one machine. The server could handle the compute, but the network plan couldn't survive my workflow.

When the suspension email came through, my client's friend Mr. Rohini Kumar Barla — the owner of ChemSus Technologies — raised a support ticket with RackNerd. The response confirmed it: bandwidth limit exceeded, server paused. But this wasn't a billing issue to fix with $7. This was a wake-up call about how I was structuring my entire workflow.

Starting Over — Two People, Two Responsibilities

My client's friend Mr. Rohini Kumar Barla provisioned a new server and also retained control of the domain configuration, since chemsus.in was registered under his Namecheap account. This time, we divided the work clearly: I would handle the full server-side deployment — Node.js, PM2, Nginx, SSL, firewall — and Mr. Barla would handle the DNS management on Namecheap as well as the server provisioning.

This separation wasn't just about convenience. It reflected a real-world production workflow: the developer manages the application infrastructure, and the domain owner manages DNS. Neither steps on the other's work, and both can verify independently that their side is functioning correctly.

⚡ Developer — Pavan Kumar Ponnuri

Set up the production server from scratch: installed Node.js 20 via NVM, cloned the project from GitHub, configured environment variables, initialized the SQLite database, installed PM2 for process management, set up Nginx as a reverse proxy on port 5656, configured SSL with Certbot, and opened firewall ports with UFW.

Development now happens locally on an Ubuntu VM. Code is pushed to GitHub, then pulled to the production server only when ready for deployment.

🌐 Domain Owner — Mr. Rohini Kumar Barla

Provisioned the new production server and logged into Namecheap to configure the DNS A records — pointing both the root domain (chemsus.in) and the www subdomain to the new production server's IP address. Used a Claude thread to understand the exact DNS configuration needed and get step-by-step guidance.

After updating the records, verified global DNS propagation using whatsmydns.net — confirming the A records had propagated to DNS servers worldwide and the domain was resolving correctly to the new server IP.

What made this collaboration interesting is that both sides used AI independently to get the job done. I used Claude CLI and Claude chat to generate a structured deployment checklist and troubleshoot server issues in real-time. Mr. Barla — who isn't a developer — opened a separate Claude conversation to walk through the Namecheap DNS configuration, getting guidance on which record types to set, what values to enter, and how to verify the result. Two people, two Claude threads, one production system coming together.

Why this matters

This wasn't two developers coordinating over Slack. It was a developer and a non-technical client, each using AI independently to own their part of a production deployment. Mr. Barla didn't need to understand Nginx or SSH. I didn't need access to his Namecheap account. Claude bridged the knowledge gap on both sides — and whatsmydns.net gave us the shared verification point to confirm everything was wired up correctly.

The Rebuild: From Clone to Live

Node.js via NVM

The first lesson from my earlier deployment still applied: never rely on the system-installed Node.js. The previous server had an outdated version that choked on modern JavaScript syntax like the nullish coalescing operator (??). NVM solved this cleanly — install it, pull Node 20, set it as the default, and move on. 

curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
source ~/.bashrc
nvm install 20 && nvm alias default 20

Application Setup

With the project cloned from GitHub, it was a matter of npm install for dependencies, npm run seed to initialize the SQLite database, and configuring the .env file with the necessary secrets — JWT keys, SMTP credentials, admin configuration, and the database path. I generated cryptographic secrets using Node's built-in crypto module rather than inventing passwords by hand. 

node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

PM2 for Process Management

On my first deployment, I had tried running the app with npm run dev, which spins up Nodemon — a tool designed purely for development hot-reloading. In production, you need PM2. It runs the application in the background, restarts it automatically on crashes, and survives server reboots through a system-level init script generated by pm2 startup. 

pm2 start backend/server.js --name chemsus
pm2 save
pm2 startup

Nginx as Reverse Proxy

The Node app runs on port 5656 internally. Users should just hit chemsus.in and see the site — no port numbers. Nginx sits in front, listens on ports 80 and 443, and forwards requests to localhost:5656. It also handles HTTP-to-HTTPS redirects and adds proper headers for WebSocket support.

A lesson burned into memory from the first deployment: always remove the default Nginx config from sites-enabled. If you don't, visitors see the "Welcome to Nginx" page instead of your application.

SSL with Certbot

Let's Encrypt via Certbot remains one of the most satisfying parts of any deployment. One command, and your site goes from the browser's "Not Secure" warning to a green padlock. Certbot also modifies your Nginx config automatically for HTTPS redirects and sets up auto-renewal so you don't wake up three months later to an expired certificate. 

sudo certbot --nginx -d chemsus.in -d www.chemsus.in

Firewall, DNS, and Verification

On my end, UFW was configured to allow SSH, ports 80 and 443 for web traffic, and 5656 for the internal app. On Mr. Barla's end, the A records were configured in Namecheap — pointing both @ (root domain) and www to the new server's IP address.

Mr. Barla then verified DNS propagation using whatsmydns.net — a tool that queries DNS servers across the globe and visually shows whether your A record has reached everywhere. When every node on the map turns green, you know the domain is resolving correctly worldwide. This was the final handshake — server infrastructure and domain routing, connected and verified independently by both sides.

The Separation That Changed Everything

The architecture of the app didn't change between the first and second deployment. The Nginx config was nearly identical. PM2 was the same. What changed was the workflow around it — and the clear division of responsibilities between developer and client.

Now, development happens on my local Ubuntu VM. I write code, test it, commit to GitHub. When a feature is ready for production, I SSH into the server, pull the latest changes, and restart PM2. The production server's bandwidth is consumed only by real user traffic — not by my development toolchain downloading half the npm registry or streaming AI responses.

Key Lesson

A production server should serve production traffic. That's it. The moment you start developing on it — installing dev dependencies, running AI tools, experimenting with packages — you're treating a production environment like a playground. The bandwidth bill was just the symptom. The missing architecture was the disease.

What This Experience Actually Taught Me

Tutorials teach you how to deploy an app. They don't teach you what happens when your deployment falls apart and a client is asking why their website is down. The real learning happens in the gap between "it works on my machine" and "it works in production, reliably, under real constraints."

  • Separate your environments ruthlessly. Development, staging, and production are different concerns with different resource profiles. Mixing them will always cost you — in bandwidth, in stability, or in client trust.
  • Understand your hosting plan's limits. "Generous bandwidth" doesn't mean unlimited. Read the fine print. Monitor your usage. Set alerts if the provider supports them.
  • AI-assisted development is powerful but network-hungry. Tools like Claude CLI are extraordinary for productivity, but they generate significant traffic. That's free on your local machine. On a metered VPS, it's a liability.
  • Non-technical stakeholders can own their part. Mr. Rohini Kumar Barla configured DNS independently using a Claude thread and verified the result himself on whatsmydns.net — no developer hand-holding required. AI lowers the barrier to infrastructure work for everyone.
  • Document your deployment before you need to repeat it. When I rebuilt from scratch, having a structured record of the first deployment made the second one take hours instead of days.
  • PM2, Nginx, and Certbot are non-negotiable. Every Node.js deployment I do now starts with these three. They're the infrastructure equivalent of seatbelts — you don't skip them because the road looks smooth.
  • GitHub is the bridge between environments. Code moves from local to production via commits and pulls — never by editing files on the server over SSH.
• • •

Looking Forward

chemsus.in is live. The padlock is green. PM2 is running in the background. The production server is lean — it runs the app and nothing else. All the creative chaos of development stays on my local Ubuntu VM, where bandwidth is free and mistakes don't take down a client's website.

The next steps are clear: set up a CI/CD pipeline so deployments from GitHub to the server are automated, add monitoring and logging to catch problems before they become outages, and implement rate limiting and security headers to harden the production environment. But those are optimizations. The foundation — the separation of environments, the clean deployment pipeline, the collaborative workflow between developer and client, each using AI to own their piece — is already solid.

Sometimes the best lessons come from the most expensive mistakes. Mine cost 2 terabytes of bandwidth and a few stressful days. What I got back was an understanding of production infrastructure that no tutorial could have given me — and a workflow where both the developer and the client contribute confidently to a shared production system, with AI making both sides more capable than they'd be alone.
